# Pre-commit Workflow

Перед каждым коммитом выполнить:

## 1. Проверить изменения
```bash
git status
git diff --name-only
```

## 2. Обновить файл прогресса

В зависимости от того, какой раздел был изменён:

| Раздел | Файл прогресса |
|--------|----------------|
| JavaScript (BFE) | `progress/js.md` |
| Algorithms | `progress/algorithms.md` |
| CSS | `progress/css.md` |
| React | `progress/react.md` |

### Что заполнить в таблице:
- **Статус**: ✅ (с первого раза) / ⚠️ (с ошибками) / ❌ (не решил)
- **Повторить**: дата через неделю если были ошибки
- **Дата**: текущая дата
- **Комментарий**: краткое описание ошибок или важных моментов

## 3. Сделать коммит
// turbo
```bash
git add -A && git commit -m "тип: описание"
```

### Типы коммитов:
- `feat`: новая задача решена
- `fix`: исправление ошибки в решении
- `docs`: обновление документации
- `refactor`: улучшение кода без изменения логики

---

## 4. Правила оформления решённых задач

### Улучшение читаемости:
- Переименовать переменные для лучшей читаемости
- Использовать понятные имена: `currentDepth` вместо `d`, `element` вместо `el`
- Добавить комментарии к неочевидным местам

### Best Solution (если отличается от моего):
- Добавить секцию "ЛУЧШЕЕ РЕШЕНИЕ" с альтернативным подходом
- Объяснить почему оно лучше (читаемость, производительность, лаконичность)
- Это повышает насмотренность и расширяет арсенал решений
